## 二、Phase 1：多路召回系统

### 2.1 BM25 关键词召回（新增）

**文件位置**：`backend/app/rag/retriever/bm25_retriever.py`

**功能需求**：
- 基于 `rank_bm25` 库实现 BM25 关键词检索
- 接收 query 和文档列表，返回 BM25 得分排序结果
- 配置项：BM25 参数 k1=1.5, b=0.75

**接口定义**：
```python
class BM25Retriever:
    def __init__(self, k1=1.5, b=0.75)
    def fit(self, documents: list[Document])  # 构建 BM25 索引
    def retrieve(self, query: str, k: int = 4) -> list[tuple[Document, float]]
```

### 2.2 混合检索融合器（新增）

**文件位置**：`backend/app/rag/retriever/hybrid_retriever.py`

**功能需求**：
- 融合语义检索（ChromaDB cosine）和 BM25 关键词检索
- 支持可配置权重：`alpha`（语义权重，默认0.7）+ `beta`（关键词权重，默认0.3）
- 分数归一化 → 加权融合 → 去重排序
- 兜底策略：单路为0时自动切到另一路

**融合公式**：
```
final_score = alpha * norm(semantic_score) + beta * norm(bm25_score)
```

### 2.3 Cross-encoder 精排（新增）

**文件位置**：`backend/app/rag/retriever/reranker.py`

**功能需求**：
- 基于 `sentence-transformers` 的 Cross-encoder 模型（默认 `BAAI/bge-reranker-v2-m3`）
- 对召回结果（Top-K）做精细重排序
- 返回重新排序后的文档列表 + 精排分数
- **降级策略**：模型加载失败时跳过精排，直接返回召回结果

**接口定义**：
```python
class CrossEncoderReranker:
    def __init__(self, model_name: str, device: str = "cpu")
    def rerank(self, query: str, documents: list[Document], top_k: int = 4) -> list[Document]
```

### 2.4 集成到 RAG Pipeline

**修改文件**：`backend/app/rag/pipeline.py`

- 在 RAGPipeline 中新增 `hybrid_retrieve()` 方法
- 调用链：hybrid_retrieve → rerank → format_context
- 保持原有纯语义检索接口的向下兼容
- 通过配置开关 `RETRIEVER_MODE=hybrid|semantic` 切换

### 2.5 验收边界

| 验收项 | 预期结果 |
|-------|---------|
| BM25 对精确术语（如具体表名、列名）召回率高于纯语义 | Recall@4 提升 ≥5% |
| 混合检索在电商知识库上 MRR 提升 ≥3% | MRR ≥ 0.85 |
| Cross-encoder 精排后 Top-1 准确率提升 | Top-1 Acc 提升 ≥5% |
| 任意组件加载失败时系统不崩溃 | 降级为纯语义检索 |
| 响应时间增加不超过 200ms | 单次查询 ≤ 2s |

## 三、Phase 2：评估体系

### 3.1 金标数据集（新增）

**文件位置**：`backend/data/evaluation/benchmark.json`

**数据格式**：
```json
[
  {
    "id": "eval_001",
    "question": "上个月销量最高的产品是什么？",
    "expected_sql": "SELECT p.name, SUM(o.quantity) as total_qty FROM products p JOIN orders o ON p.id = o.product_id WHERE o.order_date >= '2026-06-01' AND o.order_date < '2026-07-01' GROUP BY p.id, p.name ORDER BY total_qty DESC LIMIT 1",
    "expected_answer_contains": ["销量最高", "第一名"],
    "relevant_doc_ids": ["doc_orders_001", "doc_products_002"],
    "category": "sql_generation"
  }
]
```

**规模**：不少于 20 条，覆盖如下分类：
- SQL 生成类：10 条（单表查询、多表 JOIN、聚合、窗口函数等）
- RAG 检索类：5 条（需要召回特定知识）
- 混合类：5 条（需要知识检索 + SQL 生成）

### 3.2 指标计算模块（新增）

**文件位置**：`backend/app/evaluation/metrics.py`

**实现指标**：

| 指标 | 公式 | 说明 |
|-----|------|------|
| Recall@K | relevant_retrieved / total_relevant | Top-K 召回率 |
| Precision@K | relevant_in_top_k / K | Top-K 精确率 |
| MRR | 1/rank_of_first_relevant | 平均倒数排名 |
| NDCG@K | DCG@K / IDCG@K | 归一化折损累计增益 |
| Top-1 Accuracy | correct_top1 / total_queries | 首位命中率 |

### 3.3 对比运行器（新增）

**文件位置**：`backend/app/evaluation/runner.py`

**功能需求**：
- 支持三种检索策略对比：纯语义 / BM25 / 混合
- 自动计算每种策略下的所有指标
- 输出对比表格（JSON 格式）
- 可以通过 API `/api/evaluation/run` 触发

### 3.4 评估 API（新增）

**文件位置**：`backend/app/routes/evaluation_v2.py`

- `GET /api/evaluation/stats` → 评测集概况
- `POST /api/evaluation/run` → 执行完整评测
- `GET /api/evaluation/results/{eval_id}` → 查看某次评测结果

### 3.5 验收边界

| 验收项 | 预期结果 |
|-------|---------|
| 20+ 条评测数据可加载 | 全部通过 schema 校验 |
| 四种召回策略均可独立评测 | 返回完整指标 |
| 前端可查看评测结果对比 | 可视化对比表格 |
| 指标计算数学正确 | 手动计算结果一致 |

## 四、Phase 3：Embedding 模型微调

### 4.1 训练数据准备（新增）

**文件位置**：`backend/scripts/prepare_training_data.py`

**功能需求**：
- 从 `nl2sql_knowledge.json` 提取 Question-SQL 对作为训练数据
- 构建对比学习样本：(query, positive_doc, negative_doc) 三元组
- negative 采样策略：同一类别内不相似样本 / 随机采样
- 输出：`data/training/train_triplets.json`

### 4.2 微调训练脚本（新增）

**文件位置**：`backend/scripts/finetune_embedding.py`

**功能需求**：
- 基于 `sentence-transformers` 的 `MultipleNegativesRankingLoss`
- 加载 `BAAI/bge-large-zh-v1.5` 作为基座模型
- 支持配置：batch_size=8, epochs=3, learning_rate=2e-5
- 训练完成后保存模型到 `data/models/finetuned-bge`
- 支持 resume 和 early stopping

### 4.3 微调前后对比（新增）

**文件位置**：`backend/scripts/evaluate_embedding.py`

**功能需求**：
- 加载微调前 / 微调后两个模型
- 在评测集上计算 Recall@K 和 MRR
- 输出对比报告

### 4.4 集成到配置系统

**修改文件**：`backend/app/config.py`

- 新增 `EMBEDDING_MODEL_FINETUNED` 配置项
- 当微调模型存在时自动优先使用

### 4.5 验收边界

| 验收项 | 预期结果 |
|-------|---------|
| 训练数据 ≥ 50 条有效三元组 | 产出 train_triplets.json |
| 微调脚本可完整运行 3 个 epoch | 无报错，模型权重保存成功 |
| 微调后模型在电商领域 Recall@4 提升 ≥ 3% | 对比报告明确显示提升 |

## 五、Phase 4：Agent 记忆增强

### 5.1 对话摘要模块（新增）

**文件位置**：`backend/app/memory/summarizer.py`

**功能需求**：
- 使用 LLM 对历史对话生成结构化摘要
- 摘要格式：
```json
{
  "user_intent": "查询上个月销售数据",
  "key_entities": ["产品A", "2026年6月", "销量"],
  "sql_executed": "SELECT ...",
  "important_facts": ["产品A上月销量100件"],
  "user_preferences": ["偏好柱状图展示"]
}
```
- 每 5 轮对话自动触发一次摘要
- 摘要存储到 SQLite `memory_summaries` 表

### 5.2 长期记忆存储（新增）

**文件位置**：`backend/app/memory/store.py`

**功能需求**：
- 基于 ChromaDB 存储对话摘要向量
- 每条摘要作为一个 Document 存入
- 元数据标记：`conversation_id`, `timestamp`, `turn_count`
- 支持按时间衰减权重检索

### 5.3 记忆感知检索（新增）

**文件位置**：`backend/app/memory/retriever.py`

**功能需求**：
- 当前用户提问时，同时检索：
  1. RAG 知识库（领域知识）
  2. 对话记忆库（历史上下文）
- 记忆检索结果格式化后注入 System Prompt
- **时间衰减**：近 1 小时的记忆权重 1.0，1-24 小时 0.7，超过 24 小时 0.3

### 5.4 集成到 Agent

**修改文件**：`backend/app/agent/prompts.py`

- 在 System Prompt 中新增记忆上下文段落
- 格式：
```
【历史对话记忆】
[摘要1]: 用户上次查询了产品销量
[摘要2]: 用户偏好柱状图展示
```

### 5.5 验收边界

| 验收项 | 预期结果 |
|-------|---------|
| 5 轮对话后自动生成摘要 | SQLite 中出现摘要记录 |
| 新对话中检索到历史记忆 | Agent 回答引用历史信息 |
| 时间衰减生效 | 旧记忆权重低于新记忆 |
| Agent 响应中包含记忆上下文 | System Prompt 中出现【历史对话记忆】段落 |

## 六、Phase 5：CI/CD 与云部署

### 6.1 GitHub Actions 工作流（新增）

**文件位置**：`.github/workflows/deploy.yml`

**触发条件**：
- push 到 `main` / `master` 分支
- 手动触发 (workflow_dispatch)

**流水线步骤**：
1. Checkout 代码
2. 设置 Python 3.10 / Node 20 环境
3. 安装后端依赖
4. 安装前端依赖并构建
5. 运行测试（单元测试 + 集成测试）
6. 构建 Docker 镜像
7. 推送镜像到 Docker Hub / 阿里云镜像仓库
8. SSH 到云服务器拉取镜像并重启

### 6.2 Docker 优化

**修改文件**：`backend/Dockerfile`

**优化内容**：
- 多阶段构建，减小镜像体积
- 预下载 Embedding 模型到镜像中（避免启动时下载）
- 设置健康检查端点 `/api/health`
- 添加依赖缓存层优化构建速度

### 6.3 云部署配置

**新增文件**：`deploy/docker-compose.prod.yml`

**内容**：
- 后端服务（1 副本）
- 前端服务（1 副本）
- Nginx 反向代理（SSL 自动续签）
- 持久化数据卷

**所需环境变量**（云端配置）：
```bash
# .env.production
DEEPSEEK_API_KEY=xxx
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-chat
EMBEDDING_MODEL=BAAI/bge-large-zh-v1.5
SERVER_IP=your_server_ip
SERVER_USER=root
```

### 6.4 自动部署脚本

**新增文件**：`deploy/deploy.sh`

**部署流程**：
1. GitHub Actions 构建 Docker 镜像
2. 通过 SSH 将 docker-compose.prod.yml 传输到服务器
3. 服务器拉取最新镜像
4. 停止旧容器，启动新容器
5. 健康检查确认服务正常
6. 如果检查失败则回滚到上一个版本
